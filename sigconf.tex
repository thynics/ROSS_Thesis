\begin{abstract}
In multi-GPU task-graph runtimes, placement determines both current execution time and the residency seen by later tasks. HEFT-style earliest-finish-time placement accounts for the current task's input communication, but a locally optimal placement can move a mutable object away from a useful owner and increase later communication. We identify a model-level opportunity: an alternative may finish the current task later yet leave unchanged the maximum modeled completion time after appending that task to the already committed schedule.

We propose prefix-safe copy-aware placement (P), a lightweight refinement of HEFT-style device selection. It admits a lower-copy alternative only when the resulting one-step modeled prefix-completion scalar stays within numerical tolerance of the HEFT decision. This is not a guarantee for unscheduled successors or final makespan. We implement the scheduler in CUDASTF and evaluate five benchmarks on eight NVIDIA L4 GPUs. On two preselected workload sizes where placement changes, P reduces scheduler-modeled copy by 39--81\% and improves latency by 1.18--1.31\(\times\) over the runtime-default HEFT path; against a framework-matched identity reference, the corresponding gains are 1.16--1.27\(\times\). Three unchanged workloads act as timing controls, and exact gate removal exposes a workload-dependent safety--opportunity trade-off. A separate same-build, one-run diagnostic finds exact agreement between modeled and CUDASTF-issued peer-copy events and bytes in all ten treatment--workload executions, and workload-dependent ordering against a port of the antecedent fixed gate.
\end{abstract}

\section{Introduction}\label{sec:introduction}

In a multi-GPU runtime, the scheduler repeatedly assigns ready tasks to devices while the runtime manages the movement of their input and output data. These two responsibilities are tightly coupled: a placement decision determines not only when the current task can execute, but also where its data will be available for subsequent tasks. Effective placement must therefore account for both device availability and data locality. A widely used strategy is \emph{HEFT-style earliest-finish-time placement}. For each ready task, the scheduler estimates its finish time on every candidate device using device readiness, input-data availability, communication cost, and predicted execution time, and then selects the device with the earliest finish~\citep{topcuoglu2002heft}. We use this communication-aware device-selection rule as the baseline. In particular, it already accounts for the communication required to execute the current task.

Communication, however, affects execution beyond the current placement decision. In runtimes with software-managed data residency, executing a task on a device may change where the updated versions of its logical data objects reside. A placement that minimizes the current task's earliest finish time can therefore leave the runtime in a data-residency state that increases the communication required by later dependent tasks. We refer to this general interaction as a \emph{placement--residency conflict}.

A simple migrate--return pattern illustrates the conflict. Consider a mutable object currently resident on one GPU. A read--write task may finish slightly earlier on another GPU after migrating that object. Once the task completes, the updated object resides on the selected GPU. If a later dependent task executes on the original GPU, the updated object must be transferred back. The first placement is locally optimal under the current earliest-finish estimate, yet it induces additional successor communication. Repeated instances of this pattern can amplify a small local placement advantage into a larger data-movement and readiness penalty along a dependency chain.

The key observation of this work is narrower. A placement alternative may complete the current task later than the HEFT choice while its modeled finish remains below the ready-time frontier created by previously scheduled work. The alternative then leaves unchanged a one-step scalar: the maximum modeled completion time after appending the current task to the committed schedule. This slack can support a locality refinement, but it does not imply that an unscheduled successor or the final application will be undelayed.

Our prior thesis evaluated a HEFT-relative residual gate that admitted lower-copy alternatives under a normalized current-task EFT penalty and introduced the criticality proxy reused here~\citep{chengli2026multiobjective}. This paper does not claim copy-aware residual placement as new. Its algorithmic increment is to replace that penalty with a translation-invariant frontier test whose exact one-step model property can be stated directly, and its experimental increment is to isolate that test with an exact gate-removal control and counterbalanced execution.

Based on this observation, we propose prefix-safe copy-aware placement (P). Rather than replacing HEFT-style device selection, it first computes the HEFT placement and then evaluates lower-current-copy alternatives. It accepts an alternative only if it satisfies the frontier test and a copy-gain threshold; otherwise, it returns the baseline. We call the rule prefix-safe solely with respect to the modeled one-step quantity formalized in Section~\ref{sec:prefix-admission}.

In summary, this work makes the following contributions:
\begin{itemize}
\tightlist
\item
  We formalize the one-step placement--residency trade-off and delimit the supported scope to owner-local alternatives observable from current scheduler state, without successor prediction.
\item
  We propose a frontier-based refinement that admits lower-current-copy alternatives while preserving, within numerical tolerance, the modeled maximum completion after the current task is appended. We state the guarantee and its non-guarantees explicitly.
\item
  We evaluate a CUDASTF prototype using counterbalanced order, a framework-matched identity reference, locality-aware baselines, and exact gate removal. On two preselected opportunity configurations, P reduces modeled copy by 39--81\% and improves latency by 1.16--1.27\(\times\) over placement-identical I; three unchanged workloads bound timing variation. A common-build diagnostic validates modeled peer-copy accounting against runtime-issued copies and directly exposes the mixed trade-off against the antecedent gate.
\end{itemize}

\section{Background and Motivation}\label{sec:background}

\subsection{Runtime Model}\label{sec:runtime-model}

We consider a task-graph runtime that schedules a directed acyclic graph (DAG) \(G=(V,E)\) onto a set of GPUs \(D\). Each vertex \(v\in V\) represents a task, and each edge denotes a data dependency. For every task--device pair \((v,d)\), the runtime maintains a predicted execution time \(p_v(d)>0\). Tasks access data through logical objects. Each object \(x\) has size \(\operatorname{bytes}(x)\), and the runtime estimates its transfer time as
\[
c_x=\frac{\operatorname{bytes}(x)}{B},
\]
where \(B\) is the effective bandwidth assumed by the runtime's transfer-cost model. The quantity \(c_x\) is therefore a scheduling estimate rather than a measurement of instantaneous interconnect bandwidth.

Our prototype is implemented in CUDASTF~\citep{augonnet2024cudastf}, which tracks logical-data residency using software-managed Modified/\allowbreak Shared/\allowbreak Invalid (MSI) metadata. We adopt the same abstraction in the model: \(M(x,d)\in\{M,S,I\}\) records the state of object \(x\) on device \(d\), where \(M\) and \(S\) indicate that \(d\) holds a valid instance of the current version, whereas \(I\) indicates that the local instance is invalid. If a task requires \(x\) on a device where \(M(x,d)=I\), the runtime must obtain the current version from a device that holds a valid instance before the task can execute. After a write-like access to \(x\) on device \(d\), the newly produced version becomes Modified on \(d\), and conflicting instances on the remaining devices become Invalid.

The scheduler also maintains a modeled ready time \(L(d)\) for each device, defined as the earliest time at which \(d\) can begin newly assigned work under the schedule constructed so far. Because tasks are placed one at a time, \(L(d)\) and \(M(x,d)\) always refer to this evolving state at the current decision. The placement decision is thus based on two forms of runtime state: \(L(d)\) captures device availability, while \(M(x,d)\) captures the availability of each task input.

\subsection{HEFT Baseline}\label{sec:heft-baseline}

Let \(A(v,d)\) denote the latest time at which all read and read--write inputs of task \(v\) are available on device \(d\). If a required input is not locally available, \(A(v,d)\) includes the time needed to transfer its current version to \(d\). The HEFT baseline places task \(v\) on the device with the earliest modeled finish time:
\[
F(v,d)=\max\{L(d),A(v,d)\}+p_v(d),
\qquad
b=\arg\min_{d\in D}F(v,d),
\]
where \(b\) is the device selected by the HEFT baseline; ties are broken by lower device ID.

Our prototype implements only HEFT-style \emph{device selection}; it does not implement HEFT's upward-rank task-ordering stage. The harness requests no CUDASTF task reorderer, so deferred tasks are presented sequentially in deterministic benchmark-construction order, each placement is irrevocable, and CUDA dependencies determine execution readiness.

\subsection{Motivation}\label{sec:motivation}

The HEFT baseline minimizes the modeled finish time of the current task, but it does not explicitly account for how the placement changes data residency for later tasks. After a write-like access, the updated version of a data object resides on the selected device, while conflicting copies on the remaining devices become invalid. Consequently, a placement that is locally optimal for the current task may increase the communication required by its successors. We refer to this interaction as a \emph{placement--residency conflict}.

This conflict can arise even in a simple two-GPU example. Suppose the current version of a large object \(x\) resides on GPU 1, and placing a read--write task \(v\) on GPU 0 shortens its modeled finish time by a small amount \(\epsilon\). The HEFT baseline therefore selects GPU 0. After \(v\) completes, the updated version of \(x\) now resides on GPU 0. If a dependent task later executes on GPU 1, the runtime must transfer the updated object back before that task can begin. This additional transfer would not have been required had \(v\) remained on GPU 1. In other words, a small improvement in the current task's finish time can introduce substantially more communication for later tasks.

This example shows that minimizing the current task's modeled finish time does not necessarily minimize communication in the remaining execution. The proposed gate nevertheless observes only the current task's read and read--write inputs; it does not predict successor use. It therefore targets the owner-local/migrate--return subclass in which reducing current input movement also improves the resulting mutable-data residency, rather than every placement--residency conflict.

\section{Scheduler Design}\label{sec:scheduler-design}

Given the next task \(v\), the scheduler first computes the HEFT-style device and then screens lower-copy alternatives through the prefix-admission and copy-gain gates in Figure~\ref{fig:prefix-safe-selector}.

\subsection{Inputs and HEFT baseline}\label{sec:inputs-baseline}

For each candidate device \(d\in D\), the scheduler evaluates four quantities. The HEFT model provides the predicted finish time \(F(v,d)\) and the device ready time \(L(d)\). The modeled communication cost \(K(d)\) is the total transfer time of the read and read--write inputs required by \(v\) on \(d\). The modeled queue delay is
\[
Q(d)=\max\{0,L(d)-A(v,d)\},
\]
where \(A(v,d)\) is the time at which all required inputs are available on \(d\). Thus, \(K(d)\) captures the communication required by the placement, whereas \(Q(d)\) captures the delay caused by device availability after the inputs have become ready.

The HEFT baseline device is
\[
b=\arg\min_{d\in D}F(v,d).
\]
Let
\[
R=\max_{d\in D}L(d)
\]
denote the current device-ready frontier. We define the modeled prefix completion under the HEFT baseline and under a candidate \(d\) as
\[
P_b=\max\{R,F(v,b)\},
\qquad
P(d)=\max\{R,F(v,d)\}.
\]

\subsection{Prefix Admission}\label{sec:prefix-admission}

A non-baseline device \(d\ne b\) is \emph{prefix-admissible} when
\[
P(d)\le P_b+\varepsilon_p,
\]
where \(\varepsilon_p\) is a numerical tolerance.

\paragraph{Proposition 1 (one-step modeled prefix bound).}
Fix the modeled predecision state \(L\), \(A\), and \(p_v\), and append exactly one irrevocable placement of \(v\). Replacing baseline \(b\) by any prefix-admissible \(d\) increases the resulting scalar \(\max\{\max_{j\in D}L(j),F(v,\cdot)\}\) by at most \(\varepsilon_p\).
\emph{Proof.} The scalar is \(P_b\) under \(b\) and \(P(d)\) under \(d\); the admission inequality gives \(P(d)-P_b\le\varepsilon_p\).

The proposition is a property of the fixed scheduler model after one decision. It does not bound the EFT of an unscheduled successor, a later prefix, actual wall time under model error or resource contention, or final makespan. Here \(\varepsilon_p\) handles floating-point comparison only; it is not an uncertainty margin for prediction error.

\subsection{Copy Gate and Ranking}\label{sec:copy-gate}

Prefix admission bounds the modeled timing effect of an alternative; the copy gate determines whether the alternative provides sufficient communication benefit. Relative to the HEFT baseline, the scheduler computes the normalized copy gain
\[
G_K(d)=\frac{K(b)-K(d)}{\max\{K_{\min},K(b)\}}
\]
and the normalized queue-delay gain
\[
G_Q(d)=\frac{Q(b)-Q(d)}{\max\{Q_{\min},Q(b)\}}.
\]

A candidate passes the copy gate when it yields a positive copy saving and satisfies
\[
G_K(d)\ge \tau(v).
\]
The threshold \(\tau(v)\) is conditioned on a task-difficulty score. Here \(\hat p_v\) is the calibrated GPU-0 median task cost in milliseconds, shared across the homogeneous GPUs; \(n_v=|\mathrm{deps}(v)|\) counts all declared dependency entries; and \(B_v\) is the byte sum of read and read--write inputs. With
\[
\operatorname{clip}(z)=\min\{1,\max\{0,z\}\},
\]
the normalized components and aggregate score are
\[
\begin{aligned}
c_p&=\operatorname{clip}\!\left(\frac{\ln(1+\hat p_v/(1\,\mathrm{ms}))}{\ln 501}\right),
&c_n&=\operatorname{clip}\!\left(\frac{n_v}{16}\right),\\
c_B&=\operatorname{clip}\!\left(\frac{\ln(1+B_v/(1\,\mathrm{byte}))}{\ln(1+2^{30})}\right),\\
c(v)&=\operatorname{clip}(0.55c_p+0.30c_n+0.15c_B).
\end{aligned}
\]
The scheduler uses \(\tau(v)=0.25\) when \(c(v)>0.60\) and \(\tau(v)=0.15\) otherwise.

Candidates that pass both gates are ranked by
\[
S(d)=G_K(d)+\lambda_Q\max\{0,G_Q(d)\}.
\]
This score prioritizes copy reduction and provides an additional preference for candidates that also reduce queue delay. Scores within \(10^{-6}\) are tied; ties are resolved by lower \(F(v,d)\) within a \(10^{-6}\) ms tolerance and then lower device ID. If no candidate passes both gates, the scheduler returns \(b\). The implementation uses \(\varepsilon_p=10^{-6}\) ms, \(K_{\min}=Q_{\min}=0.001\) ms, and \(\lambda_Q=0.35\).

Algorithm~\ref{alg:prefix-safe} consolidates the procedure. After computing baseline \(b\) and frontier \(R\), it screens each remaining device through the one-step prefix gate and then the copy gate. Surviving candidates are ranked by \(S(d)\); the scheduler commits the highest-ranked device or retains \(b\). The runtime then updates device-ready times and residency metadata before the next placement.

\section{Experimental Methodology}\label{sec:methodology}

\subsection{Platform}\label{sec:platform}

Experiments were conducted on a single dual-NUMA node with eight NVIDIA L4 GPUs (23,034 MiB each), using driver 595.71.05, CUDA 12.4.131, GCC 13.3.0, and CUDASTF/CCCL 3.2.0.0~\citep{nvidia2026cccl320} in a C++17 Release build. GPUs 0--3 and 4--7 form two NUMA-local PCIe groups, and cross-group transfers traverse the system interconnect. Transfer time is modeled using a single copy-bandwidth parameter \(B=20.7\) GB/s, the median of a directed-pair bandwidth microbenchmark. Task compute costs are calibrated once on GPU 0 and shared across the eight identical L4s.

\begin{algorithm}[t]
\caption{Prefix-safe copy-aware placement.}
\label{alg:prefix-safe}
\begin{algorithmic}[1]
\Procedure{PrefixSafeCopy}{$\text{task }v,\text{ candidates }D$}
\State $b\gets\arg\min_{d\in D}F(v,d)$
\State $R\gets\max_{d\in D}L(d)$
\State $P_b\gets\max\{R,F(v,b)\}$
\State $\mathit{eligible}\gets\emptyset$
\ForAll{$d\in D$ with $d\ne b$}
  \State $P(d)\gets\max\{R,F(v,d)\}$
  \If{$P(d)>P_b+\varepsilon_p$}
    \State \textbf{continue} \Comment{prefix gate}
  \EndIf
  \State $\Delta K\gets K(b)-K(d)$
  \State $g_K\gets\Delta K/\max\{K_{\min},K(b)\}$
  \If{$\Delta K\le 0$ or $g_K<\tau(v)$}
    \State \textbf{continue} \Comment{copy gate}
  \EndIf
  \State $g_Q\gets(Q(b)-Q(d))/\max\{Q_{\min},Q(b)\}$
  \State $S(d)\gets g_K+\lambda_Q\max\{0,g_Q\}$
  \State $\mathit{eligible}\gets\mathit{eligible}\cup\{(d,S(d))\}$
\EndFor
\If{$\mathit{eligible}=\emptyset$}
  \State \Return $b$
\EndIf
\State \Return the $d$ with largest $S(d)$; then lower $F(v,d)$ and ID
\EndProcedure
\end{algorithmic}
\end{algorithm}

\subsection{Workloads}\label{sec:workloads}

FDTD, CG, and Cholesky are task-graph adaptations of the CUDASTF sources in CCCL v3.2.0~\citep{nvidia2026cccl320}; LU is a no-pivot tiled companion we implemented. Cholesky and LU call FP64 cuBLAS and cuSOLVER kernels~\citep{nvidia2024cublas,nvidia2024cusolver}, and MiniWeather is a six-region task-graph adaptation of the miniWeather mini-app~\citep{norman2018miniweather}. For Cholesky and LU, correctness applies three fixed \(\{-1,+1\}\) probes to the reconstructed factorization, normalizes the worst projected residual by \(\lVert A\rVert_\infty\), and checks the full output checksum; the remaining workloads use the checksums listed in Table~\ref{tab:workloads}.

Inputs are deterministic across fresh treatment processes. Cholesky uses \(A_{ij}=1/(i+j+1)+2n\,\mathbf{1}[i=j]\); LU uses diagonal \(2n+1\) and off-diagonal \(((i+3j)\bmod17=0)?0.002:0.0001\). FDTD starts from zero fields with a centered sinusoidal source, and MiniWeather uses its fixed cosine-squared warm bubble. CG constructs one symmetric tridiagonal matrix per fresh process with \texttt{drand48()} and no explicit \texttt{srand48()}, hence the libc-default sequence is repeated; its 40 contexts reuse that matrix, with all-ones right-hand side and initial iterate. The scheduler and Williams order use no randomness.

The workloads are divided into two categories according to whether they expose placement opportunities. Cholesky and LU expose substantial modeled-copy opportunity and yield different H/P/N placement digests; their sizes were fixed after a single exploratory run to exhibit the mechanism. FDTD, CG, and MiniWeather yield identical H/P/N placement digests and copy totals and serve as timing controls. This selected matrix tests behavior after opportunity screening, not opportunity prevalence.

\subsection{Schedulers and Protocol}\label{sec:schedulers-protocol}

The primary experiment compares six variants. H is runtime-default HEFT-style device selection. Let \(J_\lambda(v,d)=F(v,d)+\lambda K(d)\), where \(F\) and \(K\) are in milliseconds and \(\lambda\) is dimensionless. I uses the weighted-selection implementation with \(\lambda=0\), reproducing H's placement and modeled copy. W uses \(\lambda=1\) and minimizes \(J_\lambda\) over all GPUs; objectives within \(10^{-6}\) ms tie by canonical lower \(F\), then lower device ID. O also uses \(\lambda=1\), but is enabled only when \(v\) has exactly one read--write dependency with one valid Modified owner \(o\); it selects \(o\) over \(b\) iff \(J_1(v,o)+10^{-6}\,\mathrm{ms}<J_1(v,b)\), otherwise retaining \(b\). P is prefix-safe and N disables only its prefix gate.

All experiments enable \path{CUDASTF_THESIS_DEFER_KERNEL_TASKS=1}, fully materializing each DAG before \texttt{ctx.submit()}; the harness requests no task reorderer, and placement remains sequential and irrevocable in construction order. For Cholesky and LU, the benchmark interval runs from a CUDA event before DAG construction to completion after deferred submission, covering construction, scheduling gaps, and GPU execution but excluding correctness.

Each workload contributes 36 executions (six treatments in each of six Williams blocks), for \(5\times6\times6=180\) executions total. The design~\citep{williams1949designs} places every treatment in each position once and every directed first-order adjacency once. Before the matrix, the harness performs one fresh GPU-0 calibration per workload and reuses it for all 36 executions. For each task key it records ten reservations, discards reservation 0, and takes the median of reservations 1--9. Quality gates require at least 7/9 retained samples within \(\max(25\%\text{ of median},f)\) and \(1.4826\,\mathrm{MAD}\)~\citep{rousseeuw1993alternatives} at most \(\max(10\%\text{ of median},f)\), where \(f=0.025\) ms for a footprint at most 24 bytes with median below 0.050 ms and \(0.005\) ms otherwise.

Performance runs record aggregate scheduler summaries with per-task tracing disabled. Every execution passes fail-closed checks for plan, hardware, calibration, build provenance, correctness, deterministic placement/copy/checksum outputs, and H/I identity. The primary matrix was built from an \href{https://github.com/thynics/stf_ics/commit/e3cfeaf36b6a8e3c4b3a1754270aa2ecde09647a}{archived source snapshot}; the \href{https://github.com/thynics/stf_ics/blob/58cb4c57a3279d79f7b2205d5eb42a1016e53425/jobs/20260721-151200-ross-reviewer-counterbalanced-baselines-6repeat-retry1/results/summary.md}{qualified artifact} records the full source hash, run plan, raw-log inventory, calibrations, and checks. A separate-source historical block runs H/P/N on Cholesky and LU in fixed order for five repeats; it is reported only as a robustness check.

\subsection{Metrics and Statistics}\label{sec:metrics-statistics}

The primary metrics are the benchmark interval, within-block latency ratios, scheduler-modeled input-copy bytes, and the deterministic placement digest. Following normalized-benchmark practice~\citep{fleming1986geomean}, the headline for each comparison is the geometric mean of six within-block ratios, given with ranges and direction counts as descriptive statistics rather than population intervals. The Williams balance removes aliasing with additive block, position, and first-order carryover terms on log latency. In the primary matrix, copy bytes and time are model outputs. Identical placement digests and modeled-copy totals therefore imply that any residual timing difference reflects scheduler-path and system variation rather than a locality decision.

For different-placement observations, a paired geometric-mean speedup of at least 1.03\(\times\) is material, 0.97--1.03\(\times\) neutral, and below 0.97\(\times\) a regression; these predeclared engineering thresholds are descriptive and are neither inferential tests nor calibrated to each workload's noise. Modeled-copy reduction relative to HEFT is a separate mechanism signal. Identical-placement rows are timing controls and receive no algorithmic label.

The prefix gate is classified by a paired P--N comparison. P materially protects against gate removal when the geomean of \(T_N/T_P\) is at least 1.03, N shows a material conservative cost when the geomean of \(T_P/T_N\) is at least 1.03, and the gate is otherwise neutral. A materially better P together with greater N copy reduction or positive N prefix damage is an over-optimization case, and a materially better N together with greater copy reduction is a conservative gate cost.

Scheduler share is the summed per-task scheduler decision time divided by the benchmark interval, computed identically for HEFT. We report absolute scheduler time per task alongside the ratio against a predeclared 5\% engineering target for this prototype.

\subsection{Identity and Timing Controls}\label{sec:identity-controls}

The identity control verifies that I reproduces H exactly. Across all 30 workload--block pairs, H and I have identical task and dependency-check counts, placement digests, modeled-copy accounting, scheduler-context counts, and checksums. The speedup of I over H is 1.024\(\times\) (FDTD), 1.001\(\times\) (CG), 1.035\(\times\) (Cholesky), 1.018\(\times\) (LU), and 1.000\(\times\) (MiniWeather), so same-placement path effects can reach the 1.03 threshold. Using framework-matched, placement-identical I as the reference, P/I is 1.269\(\times\) (1.258--1.273, 6/6 directions) on Cholesky and 1.156\(\times\) (1.148--1.171, 6/6) on LU. Most of each P/H gain therefore persists when default H is replaced by I. P and I still use different selector routines, so exact gate attribution comes from P/N, not P/I. FDTD, CG, and MiniWeather remain mechanism-identical timing controls.

\subsection{Ablation and Instrumentation}\label{sec:ablation}

The no-prefix ablation (N) isolates the prefix gate: it retains P's baseline, candidates, gains, thresholds, score, and tie-breaks and disables only prefix admission. The criticality weights, split, and copy thresholds are inherited from the prior placement layer~\citep{chengli2026multiobjective}; \(\lambda_Q=0.35\) is the present profile, whereas the thesis residual score evaluated queue weight zero. Every workload shares the parameters. Each decision logs copy saving, HEFT-relative EFT damage, and prefix damage; qualification requires aggregate P prefix damage to be zero within numerical tolerance.

\subsection{Diagnostic Certificate}\label{sec:diagnostic-certificate}

The successor-aware locality certificate (SALC) is a diagnostic applied only during evaluation; the runtime scheduler makes no successor prediction. It characterizes an opportunity that prefix safety cannot exploit---a device whose current EFT extends \(P_b\) even though one known future copy is much larger---for the graph-visible case in which a successor is fixed to a device.

For producer \(v\), SALC identifies write-like output symbols read by direct successors. If successor \(u\) is explicitly fixed to GPU \(a(u)\), that GPU is an authoritative anchor. For a hypothetical producer device \(d\), a conservative one-hop modeled copy quantity is
\[
\Phi_1(d)=
\sum_{(x,a)\in U}
\mathbf{1}[d\ne a]\frac{\operatorname{bytes}(x)}{B},
\]
where \(U\) deduplicates output-symbol/anchor pairs. For HEFT baseline \(b\) and alternative \(d\), SALC computes
\[
G_{\mathrm{future}}(d)=\Phi_1(b)-\Phi_1(d),
\qquad
\Delta_F(d)=\max\{0,F(d)-F(b)\},
\]
and accepts only when
\[
C_{\mathrm{SALC}}(d)=G_{\mathrm{future}}(d)-\Delta_F(d)>\mu,
\]
for margin \(\mu\) (zero in the controlled certificate). SALC selects the positive candidate with the largest certificate, breaking ties by EFT.

The inequality is a certificate only when the producer creates the unique relevant version, the fixed successor reads that same version, and no intervening task overwrites it. It certifies a modeled one-hop edge risk, and the positive evidence reported here is the authoritative fixed-successor case.

\section{Evaluation}\label{sec:evaluation}

\subsection{Primary Results}\label{sec:primary-results}

This experiment asks whether prefix-safe placement (P) improves performance when a placement--residency opportunity is present. Among the five workloads, P departs from the HEFT baseline (H) only on Cholesky and LU (Table~\ref{tab:primary-matrix}, Figure~\ref{fig:primary-results}); the other three retain H's placement and act as timing controls.

On the two opportunity workloads, P reduces scheduler-modeled copy by 81.0\% on Cholesky (128.55 to 24.41 GB) and 38.8\% on LU (148.77 to 90.98 GB). Relative to H, latency improves by 1.313\(\times\) (1.296--1.324) and 1.177\(\times\) (1.171--1.186); relative to framework-matched I, it improves by 1.269\(\times\) (1.258--1.273) and 1.156\(\times\) (1.148--1.171). All six pairs agree in direction. These results support the two screened configurations, not a claim about opportunity prevalence.

On Cholesky, W, O, and P model 116.20, 113.46, and 24.41 GB; on LU, they model 96.28, 96.14, and 90.98 GB. P is materially faster than both controls on Cholesky and than O on LU; its 1.020\(\times\) edge over W on LU is below the engineering threshold. On CG, W changes placement without changing modeled copy and runs 1.841\(\times\) slower than P. Only the exact P/N control below attributes an outcome to prefix admission.

\emph{Timing controls.} The three unchanged workloads bound the timing noise present in the effective comparisons. FDTD, CG, and MiniWeather produce identical H, P, and no-prefix (N) placement digests and modeled-copy totals, so their P/H ratios measure scheduler-path and system variation alone, and FDTD and MiniWeather are identical across all six variants. FDTD's P/H ratios span 0.801--1.157\(\times\) under identical decisions, which quantifies the timing noise of its short benchmark interval.

\emph{Gate removal.} Removing the prefix gate (N) exposes both sides of the safety--opportunity trade-off. On Cholesky, N is 1.161\(\times\) faster than P across all six pairs and lowers modeled copy to 5.90 GB. On LU, P is instead 1.210\(\times\) faster than N across all six pairs, and N accumulates 732.3 ms of modeled-prefix damage in each run while lowering modeled copy to 3.88 GB. The same admission rule rejects an aggressive placement on Cholesky while preventing a harmful one on LU, illustrating the intended condition-dependent behavior.

\section{Related Work}\label{sec:related-work}

\textbf{Direct antecedent.} Our thesis introduced a HEFT-relative residual policy, the criticality proxy, and the same platform/workload family~\citep{chengli2026multiobjective}. That policy tolerated a bounded normalized current-task EFT penalty. The present paper's increment is the frontier-based zero-prefix-damage test, its one-step invariant, and experiments that isolate this gate. The \(C_0\) diagnostic ports the thesis's fixed default gate into the common build; it is not a replay of the complete adaptive policy. Copy-aware residual placement itself is not claimed as new.

\textbf{HEFT and bounded lookahead.} HEFT places each task on the processor of minimum earliest finish time, accounting for the communication that delays its inputs~\citep{topcuoglu2002heft}, and later work enriches this decision with information about the future graph. PEFT adds an optimistic future-cost table~\citep{arabnejad2014peft}, lookahead HEFT evaluates a placement's effect on child tasks~\citep{bittencourt2010lookahead}, and interference-aware scheduling jointly models computation- and I/O-induced flows when constraining placement~\citep{bleuse2018anticipation}. These methods improve placement by predicting or searching the unscheduled graph. Prefix-safe instead makes no successor prediction and bounds only the one-step modeled scalar of Proposition~1.

\textbf{Task runtimes and locality-aware scheduling.} Task-based runtimes couple placement with data movement, and a broad line of work makes that coupling locality-aware. StarPU's data-aware scheduler combines software-managed shared memory with transfer-aware placement~\citep{augonnet2010dataaware,augonnet2011starpu}, while PaRSEC~\citep{bosilca2013parsec}, Legion~\citep{bauer2012legion}, and CUDASTF~\citep{augonnet2024cudastf} expose abstractions for dependencies, placement, and data management. On top of such runtimes, XKaapi's DADA schedules by data affinity~\citep{bleuse2014dada}, LAHeteroprio quantifies locality's effect on task-based performance~\citep{bramas2019lahp}, MCL selects resources by locality~\citep{kamatar2020locality}, and comparative studies place HEFT beside data- and locality-aware multi-CPU/multi-GPU schedulers~\citep{lima2015strategies}. More recent systems optimize locality under GPU-memory limits~\citep{gonthier2023taming}, combine ready-task locality with eviction for GPU and out-of-core linear algebra~\citep{gonthier2025darts}, and rebalance already-mapped work across GPUs~\citep{john2024multigpu}. These schedulers pursue affinity, resource selection, or migration. Our method leaves the underlying device selector intact and applies a model-local admission test to lower-current-copy alternatives.

\textbf{Transfer attribution.} Attributing GPU data movement to program context is usually a profiling task; HPCToolkit, for instance, attributes kernels and copies to heterogeneous calling contexts~\citep{zhou2021hpctoolkit}. Our runtime-issued transfer stream adds a complementary, decision-level view: it links each requested peer transfer to the scheduler decision and logical-data identity that produced it, connecting a placement choice directly to the data movement it induces. In the common-build dense diagnostic, its aggregate GPU-peer event and byte counts agree exactly with scheduler accounting in all ten executions.

\section{Limitations}\label{sec:limitations}

\textbf{Guarantee and model error.} Prefix-safe constrains only the one-step scalar in Proposition~1 under the fixed predecision model. It does not bound successor EFT, later prefixes, actual wall time, or final makespan. The \(10^{-6}\) ms tolerance is numerical, not an error budget: task-time, bandwidth, queue-frontier, contention, and overlap errors can make the realized effect differ from the modeled bound. Evaluating prediction-error-aware margins remains future work.

\textbf{Evaluation scope.} The present evidence is limited to one platform and five workloads; the two placement-changing sizes were selected after one exploratory run, while the other three are timing controls. It therefore tests two known opportunity configurations rather than estimating prevalence. Experiments assume deferred submission on homogeneous GPUs and omit dynamic arrival, scale/topology sweeps, external runtime schedulers, and a counterbalanced repeated comparison with the complete thesis policy. The common-build \(C_0/C_{0.35}\) comparison is a fixed-order, one-run diagnostic.

\textbf{Implementation and measurement.} The transfer model uses one bandwidth and shared GPU-0 task calibration, omitting pairwise NUMA bandwidth, contention, copy--compute overlap, and capacity/eviction. Primary copy quantities are scheduler outputs; the common-build diagnostic confirms equality with CUDASTF-issued GPU-peer accounting only for the two dense configurations tested. Neither quantity is physical-link traffic. The controlled trace attributes runtime-issued copies through task/data/fixed-successor identity rather than an independent version epoch. CUPTI activity tracing can corroborate copy activities~\citep{nvidia2026cupti}; routed physical PCIe/NVLink volume additionally requires topology/link counters or equivalent measurement. Scheduler efficiency also remains future work.

\section{Conclusion}\label{sec:conclusion}

Task placement couples current finish time with later data residency. Prefix-safe placement refines HEFT-style device selection by admitting lower-current-copy alternatives under a one-step modeled prefix bound, not a successor or makespan guarantee. On two preselected placement-changing configurations, P reduces modeled copy and improves latency over H and I; three placement-identical workloads bound timing variation. Exact gate removal is conservative on Cholesky but protective on LU. A common-build diagnostic matches modeled and runtime-issued peer-copy accounting exactly, while its mixed comparison with the antecedent fixed gate rules out dominance. A controlled trace demonstrates a migrate--return case outside P's no-lookahead contract. Broader workload/scale coverage, repeated comparison with the complete antecedent policy, physical-traffic measurement, prediction-error sensitivity, and lower overhead remain open.
