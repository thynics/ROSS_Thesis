\begin{abstract}
In multi-GPU task-graph runtimes, task placement determines both execution time and data locality. HEFT-style earliest-finish-time placement already accounts for the communication required by the current task, but it does not explicitly consider how a placement changes data residency for later dependent tasks. As a result, a locally optimal placement may increase the communication required by subsequent tasks. We observe that such locality opportunities can often be exploited without delaying overall execution because the additional delay of an alternative placement may be hidden while other GPUs remain occupied by previously scheduled work. This observation exposes a bounded opportunity to improve locality without sacrificing the performance robustness of HEFT.

We propose prefix-safe copy-aware placement, a lightweight refinement of HEFT-style scheduling that evaluates lower-copy alternatives relative to the HEFT decision. The scheduler accepts an alternative only if it reduces modeled data movement without extending the modeled completion of already committed execution; otherwise, it falls back to the original HEFT placement. We implement the proposed scheduler in CUDASTF and evaluate it on five multi-GPU benchmarks running on an eight-GPU NVIDIA L4 platform. Across the observed opportunity cases, the proposed method reduces scheduler-modeled copy by 39--81\% and improves latency by 1.18--1.31\(\times\) over HEFT, while remaining behaviorally identical to HEFT when no admissible locality improvement exists. Although implemented in CUDASTF, the admission rule is applicable to HEFT/EFT-style task-graph runtimes that maintain software-managed data residency.
\end{abstract}

\section{Introduction}\label{sec:introduction}

In a multi-GPU runtime, the scheduler repeatedly assigns ready tasks to devices while the runtime manages the movement of their input and output data. These two responsibilities are tightly coupled: a placement decision determines not only when the current task can execute, but also where its data will be available for subsequent tasks. Effective placement must therefore account for both device availability and data locality. A widely used strategy is \emph{HEFT-style earliest-finish-time placement}. For each ready task, the scheduler estimates its finish time on every candidate device using device readiness, input-data availability, communication cost, and predicted execution time, and then selects the device with the earliest finish~\citep{topcuoglu2002heft}. This makes HEFT-style placement a strong baseline for online task-graph scheduling. In particular, it already accounts for the communication required to execute the current task.

Communication, however, affects execution beyond the current placement decision. In runtimes with software-managed data residency, executing a task on a device may change where the updated versions of its logical data objects reside. A placement that minimizes the current task's earliest finish time can therefore leave the runtime in a data-residency state that increases the communication required by later dependent tasks. We refer to this general interaction as a \emph{placement--residency conflict}.

A simple migrate--return pattern illustrates the conflict. Consider a mutable object currently resident on one GPU. A read--write task may finish slightly earlier on another GPU after migrating that object. Once the task completes, the updated object resides on the selected GPU. If a later dependent task executes on the original GPU, the updated object must be transferred back. The first placement is locally optimal under the current earliest-finish estimate, yet it induces additional successor communication. Repeated instances of this pattern can amplify a small local placement advantage into a larger data-movement and readiness penalty along a dependency chain.

The key observation of this work is that execution slack can create opportunities to improve data locality without sacrificing performance. A placement alternative may complete the current task slightly later than the HEFT choice while the remaining GPUs are still occupied by previously scheduled work. In this situation, the additional delay is effectively hidden and does not delay the overall execution. The scheduler can therefore exploit this hidden slack to reduce data movement while preserving application performance. This observation suggests that locality improvements should not be evaluated solely by the finish time of the current task. Instead, they should be accepted whenever the additional delay can be absorbed by work already executing elsewhere.

Based on this observation, we propose prefix-safe copy-aware placement, a lightweight refinement of HEFT-style scheduling. Rather than replacing HEFT, the scheduler first computes the standard HEFT placement as a conservative baseline and then evaluates alternative placements relative to that baseline. An alternative is considered only if it reduces the modeled data movement for the current task, and it is accepted only if replacing the HEFT decision does not extend the modeled completion time of work that has already been scheduled. Among all admissible alternatives, the scheduler selects the one with the largest modeled copy saving. Otherwise, it falls back to the original HEFT placement. We refer to this admission rule as prefix-safe because it guarantees that replacing the HEFT decision does not extend the modeled completion of the execution prefix that has already been committed.

In summary, this work makes the following contributions:
\begin{itemize}
\tightlist
\item
  We show that HEFT-style placement can create placement--residency conflicts: a placement that minimizes the current task's earliest finish time may increase the communication required by later dependent tasks.
\item
  We propose prefix-safe copy-aware placement, a lightweight refinement of HEFT that evaluates lower-copy alternatives relative to the HEFT decision and admits them only when they do not delay the completion of already committed execution. This enables the scheduler to exploit hidden execution slack to improve data locality while preserving the performance robustness of HEFT.
\item
  We prototype the proposed scheduler in CUDASTF and evaluate it through a controlled methodology featuring counterbalanced execution order, identity controls, locality-aware baselines, and exact gate-removal ablations. The evaluation demonstrates a 39--81\% reduction in scheduler-modeled copy and a 1.18--1.31\(\times\) speedup over HEFT on the workloads where placement changes, while showing that the benefits of prefix-safe placement are opportunity- and condition-dependent. Although implemented in CUDASTF, the proposed admission rule is applicable to HEFT/EFT-style task-graph runtimes with software-managed data residency.
\end{itemize}

\section{Background and Motivation}\label{sec:background}

This section establishes the setting for prefix-safe placement: the multi-GPU task-graph runtime model, the HEFT baseline that our method refines, and the placement--residency conflict that motivates its design.

\subsection{Runtime Model}\label{sec:runtime-model}

We consider a task-graph runtime that schedules a directed acyclic graph (DAG) \(G=(V,E)\) onto a set of GPUs \(D\). Each vertex \(v\in V\) represents a task, and each edge denotes a data dependency. For every task--device pair \((v,d)\), the runtime maintains a predicted execution time \(p_v(d)>0\). Tasks access data through logical objects. Each object \(x\) has size \(\operatorname{bytes}(x)\), and the runtime estimates its transfer time as
\[
c_x=\frac{\operatorname{bytes}(x)}{B},
\]
where \(B\) is the effective bandwidth assumed by the runtime's transfer-cost model. The quantity \(c_x\) is therefore a scheduling estimate rather than a measurement of instantaneous interconnect bandwidth.

Our prototype is implemented in CUDASTF, which tracks logical-data residency using software-managed Modified/\allowbreak Shared/\allowbreak Invalid (MSI) metadata. We adopt the same abstraction in the model: \(M(x,d)\in\{M,S,I\}\) records the state of object \(x\) on device \(d\), where \(M\) and \(S\) indicate that \(d\) holds a valid instance of the current version, whereas \(I\) indicates that the local instance is invalid. If a task requires \(x\) on a device where \(M(x,d)=I\), the runtime must obtain the current version from a device that holds a valid instance before the task can execute. After a write-like access to \(x\) on device \(d\), the newly produced version becomes Modified on \(d\), and conflicting instances on the remaining devices become Invalid.

The scheduler also maintains a modeled ready time \(L(d)\) for each device, defined as the earliest time at which \(d\) can begin newly assigned work under the schedule constructed so far. Because tasks are placed one at a time, \(L(d)\) and \(M(x,d)\) always refer to this evolving state at the current decision. The placement decision is thus based on two forms of runtime state: \(L(d)\) captures device availability, while \(M(x,d)\) captures the availability of each task input.

\subsection{HEFT Baseline}\label{sec:heft-baseline}

Let \(A(v,d)\) denote the latest time at which all read and read--write inputs of task \(v\) are available on device \(d\). If a required input is not locally available, \(A(v,d)\) includes the time needed to transfer its current version to \(d\). The HEFT baseline places task \(v\) on the device with the earliest modeled finish time:
\[
F(v,d)=\max\{L(d),A(v,d)\}+p_v(d),
\qquad
b=\arg\min_{d\in D}F(v,d),
\]
where \(b\) is the device selected by the HEFT baseline; ties are broken by lower device ID.

\subsection{Motivation}\label{sec:motivation}

The HEFT baseline minimizes the modeled finish time of the current task, but it does not explicitly account for how the placement changes data residency for later tasks. After a write-like access, the updated version of a data object resides on the selected device, while conflicting copies on the remaining devices become invalid. Consequently, a placement that is locally optimal for the current task may increase the communication required by its successors. We refer to this interaction as a \emph{placement--residency conflict}.

This conflict can arise even in a simple two-GPU example. Suppose the current version of a large object \(x\) resides on GPU 1, and placing a read--write task \(v\) on GPU 0 shortens its modeled finish time by a small amount \(\epsilon\). The HEFT baseline therefore selects GPU 0. After \(v\) completes, the updated version of \(x\) now resides on GPU 0. If a dependent task later executes on GPU 1, the runtime must transfer the updated object back before that task can begin. This additional transfer would not have been required had \(v\) remained on GPU 1. In other words, a small improvement in the current task's finish time can introduce substantially more communication for later tasks.

This example shows that minimizing the current task's modeled finish time does not necessarily minimize the communication experienced by the remaining execution. It motivates a placement refinement that considers locality-aware alternatives to the HEFT decision while preserving the execution progress established by the HEFT baseline.

\section{Scheduler Design}\label{sec:scheduler-design}

Given a ready task \(v\) and the runtime state defined in Section~\ref{sec:runtime-model}, the scheduler selects one device from the candidate set \(D\). It first computes the placement chosen by the HEFT earliest-finish-time rule and then considers alternatives that require less modeled data movement. An alternative is eligible only if it satisfies the prefix-admission bound and the copy-gain threshold described below. The scheduler otherwise retains the HEFT placement. Figure~\ref{fig:prefix-safe-selector} shows this decision flow, which the rest of the section defines step by step.

\subsection{Inputs and HEFT baseline}\label{sec:inputs-baseline}

For each candidate device \(d\in D\), the scheduler evaluates four quantities. The HEFT model provides the predicted finish time \(F(v,d)\) and the device ready time \(L(d)\). The modeled communication cost \(K(d)\) is the total transfer time of the read and read--write inputs required by \(v\) on \(d\). The modeled queue delay is
\[
Q(d)=\max\{0,L(d)-A(v,d)\},
\]
where \(A(v,d)\) is the time at which all required inputs are available on \(d\). Thus, \(K(d)\) captures the communication required by the placement, whereas \(Q(d)\) captures the delay caused by device availability after the inputs have become ready.

The HEFT baseline device is
\[
b=\arg\min_{d\in D}F(v,d).
\]
Let
\[
R=\max_{d\in D}L(d)
\]
denote the current device-ready frontier. We define the modeled prefix completion under the HEFT baseline and under a candidate \(d\) as
\[
P_b=\max\{R,F(v,b)\},
\qquad
P(d)=\max\{R,F(v,d)\}.
\]

\subsection{Prefix Admission}\label{sec:prefix-admission}

A non-baseline device \(d\ne b\) is \emph{prefix-admissible} when
\[
P(d)\le P_b+\varepsilon_p,
\]
where \(\varepsilon_p\) is a numerical tolerance. This condition permits an alternative whose task-level finish time is later than that of the HEFT baseline when the difference remains within the device-ready frontier. Consequently, replacing \(b\) with an admitted candidate increases the scheduler's modeled prefix-completion quantity by at most \(\varepsilon_p\).

\subsection{Copy Gate and Ranking}\label{sec:copy-gate}

Prefix admission bounds the modeled timing effect of an alternative; the copy gate determines whether the alternative provides sufficient communication benefit. Relative to the HEFT baseline, the scheduler computes the normalized copy gain
\[
G_K(d)=\frac{K(b)-K(d)}{\max\{K_{\min},K(b)\}}
\]
and the normalized queue-delay gain
\[
G_Q(d)=\frac{Q(b)-Q(d)}{\max\{Q_{\min},Q(b)\}}.
\]

A candidate passes the copy gate when it yields a positive copy saving and satisfies
\[
G_K(d)\ge \tau(v).
\]
The threshold \(\tau(v)\) is conditioned on a task-difficulty score derived from the predicted execution time \(p_v\), dependency count \(n_v\), and total bytes \(B_v\) of the task's read and read--write inputs. With
\[
\operatorname{clip}(z)=\min\{1,\max\{0,z\}\},
\]
the normalized components and aggregate score are
\[
\begin{aligned}
c_p&=\operatorname{clip}\!\left(\frac{\ln(1+p_v)}{\ln 501}\right),
&c_n&=\operatorname{clip}\!\left(\frac{n_v}{16}\right),\\
c_B&=\operatorname{clip}\!\left(\frac{\ln(1+B_v)}{\ln(1+2^{30})}\right),
&c(v)&=\operatorname{clip}(0.55c_p+0.30c_n+0.15c_B).
\end{aligned}
\]
The scheduler uses \(\tau(v)=0.25\) when \(c(v)>0.60\) and \(\tau(v)=0.15\) otherwise.

Candidates that pass both gates are ranked by
\[
S(d)=G_K(d)+\lambda_Q\max\{0,G_Q(d)\}.
\]
This score prioritizes copy reduction and provides an additional preference for candidates that also reduce queue delay. The scheduler selects the candidate with the largest \(S(d)\); score ties are resolved by lower predicted finish time. If no candidate passes both gates, the scheduler returns the HEFT baseline \(b\). The implementation uses \(\varepsilon_p=10^{-6}\) ms, \(K_{\min}=Q_{\min}=0.001\) ms, and \(\lambda_Q=0.35\).

Algorithm~\ref{alg:prefix-safe} consolidates the complete placement procedure. After computing the HEFT baseline \(b\) and the ready-time frontier \(R\), it screens each remaining device through the prefix gate and, only if that passes, the copy gate. Ordering the two gates this way checks timing safety before any copy trade-off, so an alternative is scored only when it is already known not to extend the committed prefix. Surviving candidates are ranked by \(S(d)\); the scheduler commits the highest-ranked device, or retains \(b\) when none qualifies. Once the selected device is committed, the runtime updates its device-ready times and residency metadata before scheduling the next task.

\section{Experimental Methodology}\label{sec:methodology}

This section describes the experimental protocol used to evaluate the proposed scheduler, including the execution platform, workload selection, comparison treatments, statistical methodology, and diagnostic procedures.

\subsection{Platform}\label{sec:platform}

Experiments were conducted on a single dual-NUMA node with eight NVIDIA L4 GPUs (23,034 MiB each), using driver 595.71.05, CUDA 12.4.131, GCC 13.3.0, and CUDASTF/CCCL 3.2.0.0~\citep{nvidia2026cccl320} in a C++17 Release build. GPUs 0--3 and 4--7 form two NUMA-local PCIe groups, and cross-group transfers traverse the system interconnect. Transfer time is modeled using a single copy-bandwidth parameter \(B=20.7\) GB/s, the median of a directed-pair bandwidth microbenchmark. Task compute costs are calibrated once on GPU 0 and shared across the eight identical L4s.

\begin{algorithm}[t]
\caption{Prefix-safe copy-aware placement.}
\label{alg:prefix-safe}
\begin{algorithmic}[1]
\Procedure{PrefixSafeCopy}{$\text{task }v,\text{ candidates }D$}
\State $b\gets\arg\min_{d\in D}F(v,d)$
\State $R\gets\max_{d\in D}L(d)$
\State $P_b\gets\max\{R,F(v,b)\}$
\State $\mathit{eligible}\gets\emptyset$
\ForAll{$d\in D$ with $d\ne b$}
  \State $P(d)\gets\max\{R,F(v,d)\}$
  \If{$P(d)>P_b+\varepsilon_p$}
    \State \textbf{continue} \Comment{prefix gate}
  \EndIf
  \State $\Delta K\gets K(b)-K(d)$
  \State $g_K\gets\Delta K/\max\{K_{\min},K(b)\}$
  \If{$\Delta K\le 0$ or $g_K<\tau(v)$}
    \State \textbf{continue} \Comment{copy gate}
  \EndIf
  \State $g_Q\gets(Q(b)-Q(d))/\max\{Q_{\min},Q(b)\}$
  \State $S(d)\gets g_K+\lambda_Q\max\{0,g_Q\}$
  \State $\mathit{eligible}\gets\mathit{eligible}\cup\{(d,S(d))\}$
\EndFor
\If{$\mathit{eligible}=\emptyset$}
  \State \Return $b$
\EndIf
\State \Return the $d$ with largest $S(d)$; break score ties by lower $F(v,d)$
\EndProcedure
\end{algorithmic}
\end{algorithm}

\subsection{Schedulers and Protocol}\label{sec:schedulers-protocol}

The primary experiment compares six scheduler variants. H is the runtime-default HEFT scheduler. I executes the same weighted-selection code path with zero weight, reproducing H's placement and modeled copy exactly. W minimizes the current earliest-finish-time objective augmented with modeled input-copy cost across all candidate devices, whereas O restricts this weighted objective to the HEFT baseline and the unique Modified owner. P denotes the proposed prefix-safe scheduler, and N is its no-prefix ablation with prefix admission disabled. W and O serve as internal locality-aware baselines.

All experiments enable \path{CUDASTF_THESIS_DEFER_KERNEL_TASKS=1}, fully materializing each DAG before \texttt{ctx.submit()}; placement decisions remain sequential and irrevocable. For Cholesky and LU, the benchmark interval runs from a CUDA start event before DAG construction to a completion event after deferred submission, and thus covers graph construction, scheduling gaps, and GPU execution while excluding the correctness check.

Each workload runs six repeat blocks in which all six treatments execute once. A six-sequence Williams design~\citep{williams1949designs} places every treatment in each order position once and every directed first-order adjacency once, balancing the \(5\times6\times6=180\) observations within workload for block, position, and first-order carryover effects. Each block shares one fresh GPU-0 calibration. From nine timed reservations, the estimator discards the first and takes the median of the remainder, requiring at least 7/9 samples within \(\max(25\%\text{ of median},f)\) and a robust scale \(1.4826\,\mathrm{MAD}\)~\citep{rousseeuw1993alternatives} of at most \(\max(10\%\text{ of median},f)\), where \(f=0.025\) ms for a task footprint at most 24 bytes with median below 0.050 ms and \(0.005\) ms otherwise.

Performance runs record only aggregate scheduler summaries, with per-task tracing disabled. Every execution passes fail-closed checks for the Williams plan, hardware identity, fresh calibration, build and mechanism provenance, correctness, deterministic placement/copy/checksum outputs, and H/I identity. A source-distinct historical block additionally runs H, P, and N on Cholesky and LU in fixed order for five repeats (30 executions) under the same estimator; we report it as supporting replication kept separate from the counterbalanced matrix.

\subsection{Metrics and Statistics}\label{sec:metrics-statistics}

The primary metrics are the benchmark interval, within-block latency ratios, scheduler-modeled input-copy bytes, and the deterministic placement digest. Following normalized-benchmark practice~\citep{fleming1986geomean}, the headline for each comparison is the geometric mean of six within-block ratios, given with ranges and direction counts as descriptive statistics rather than population intervals. The Williams balance removes aliasing with additive block, position, and first-order carryover terms on log latency. Because copy bytes and time are model outputs, identical placement digests and modeled-copy totals imply that any residual timing difference reflects scheduler-path and system variation rather than a locality decision.

For different-placement observations, a paired geometric-mean speedup of at least 1.03\(\times\) is material, 0.97--1.03\(\times\) neutral, and below 0.97\(\times\) a regression; these thresholds were fixed before the results were read. Modeled-copy reduction relative to HEFT is reported separately as a mechanism signal. Identical-placement rows are timing controls and receive no algorithmic label.

The prefix gate is classified by a paired P--N comparison. P materially protects against gate removal when the geomean of \(T_N/T_P\) is at least 1.03, N shows a material conservative cost when the geomean of \(T_P/T_N\) is at least 1.03, and the gate is otherwise neutral. A materially better P together with greater N copy reduction or positive N prefix damage is an over-optimization case, and a materially better N together with greater copy reduction is a conservative gate cost.

Scheduler share is the summed per-task scheduler decision time divided by the benchmark interval, computed identically for HEFT. We report absolute scheduler time per task alongside the ratio against a predeclared 5\% engineering target for this prototype.

\subsection{Identity and Timing Controls}\label{sec:identity-controls}

The identity control verifies that the implementation-only variant (I) reproduces the HEFT baseline (H) exactly. Across all 30 workload--block pairs, H and I produce identical task and dependency-check counts, placement digests, modeled-copy accounting, scheduler-context counts, and correctness checksums. Consequently, the I/H timing ratio isolates scheduler-path and residual system effects rather than algorithmic differences: 1.024\(\times\) (FDTD), 1.001\(\times\) (CG), 1.035\(\times\) (Cholesky), 1.018\(\times\) (LU), and 1.000\(\times\) (MiniWeather). Cholesky exceeds the 1.03\(\times\) materiality threshold despite identical placements, illustrating that scheduler-path overhead alone can occasionally reach the materiality threshold. Because H, P, and N produce identical placements and modeled-copy totals on FDTD, CG, and MiniWeather, algorithmic conclusions are drawn only from comparisons that produce different mechanism outputs.

\subsection{Ablation and Instrumentation}\label{sec:ablation}

The no-prefix ablation (N) is the experimental control that isolates the prefix gate. Source-matched to prefix-safe, it retains the same HEFT baseline, candidate set, copy and queue gains, thresholds, score, and tie-breaks and disables only the prefix admission rule, so a P--N comparison attributes any difference to the gate alone. The scheduler parameters \(K_{\min}=Q_{\min}=0.001\) ms, the copy-gain threshold \(\tau\) (0.25 for \(c(v)>0.60\) and 0.15 otherwise), and \(\lambda_Q=0.35\) follow the prior placement layer~\citep{chengli2026multiobjective} and are shared by every workload. Each placement decision logs the accepted copy saving, the HEFT-relative EFT damage, and the prefix damage, and verification requires the aggregate prefix damage to be zero within numerical tolerance.

\subsection{Diagnostic Certificate}\label{sec:diagnostic-certificate}

The successor-aware locality certificate (SALC) is a diagnostic applied only during evaluation; the runtime scheduler makes no successor prediction. It characterizes an opportunity that prefix safety cannot exploit---a device whose current EFT extends \(P_b\) even though one known future copy is much larger---for the graph-visible case in which a successor is fixed to a device.

For producer \(v\), SALC identifies write-like output symbols read by direct successors. If successor \(u\) is explicitly fixed to GPU \(a(u)\), that GPU is an authoritative anchor. For a hypothetical producer device \(d\), a conservative one-hop modeled copy quantity is
\[
\Phi_1(d)=
\sum_{(x,a)\in U}
\mathbf{1}[d\ne a]\frac{\operatorname{bytes}(x)}{B},
\]
where \(U\) deduplicates output-symbol/anchor pairs. For HEFT baseline \(b\) and alternative \(d\), SALC computes
\[
G_{\mathrm{future}}(d)=\Phi_1(b)-\Phi_1(d),
\qquad
\Delta_F(d)=\max\{0,F(d)-F(b)\},
\]
and accepts only when
\[
C_{\mathrm{SALC}}(d)=G_{\mathrm{future}}(d)-\Delta_F(d)>\mu,
\]
for margin \(\mu\) (zero in the controlled certificate). SALC selects the positive candidate with the largest certificate, breaking ties by EFT.

The inequality is a certificate only when the producer creates the unique relevant version, the fixed successor reads that same version, and no intervening task overwrites it. It certifies a modeled one-hop edge risk, and the positive evidence reported here is the authoritative fixed-successor case.

\section{Evaluation}\label{sec:evaluation}

This section evaluates prefix-safe placement on the counterbalanced five-workload matrix, covering its performance under placement--residency opportunities, its reproducibility and copy-model sensitivity, its scheduler overhead, and a controlled validation of the migrate--return mechanism.

\subsection{Primary Results}\label{sec:primary-results}

This experiment asks whether prefix-safe placement (P) improves performance when a placement--residency opportunity is present. Among the five workloads, P departs from the HEFT baseline (H) only on Cholesky and LU (Table~\ref{tab:primary-matrix}, Figure~\ref{fig:primary-results}); the other three retain H's placement and act as timing controls.

On the two opportunity workloads, P reduces scheduler-modeled copy by 81.0\% on Cholesky (128.55 to 24.41 GB) and 38.8\% on LU (148.77 to 90.98 GB), and improves latency over H by 1.313\(\times\) (1.296--1.324\(\times\)) and 1.177\(\times\) (1.171--1.186\(\times\)), with all six within-block pairs agreeing in direction. Across the evaluated opportunity workloads, reductions in scheduler-modeled copy consistently translate into latency improvements.

Bounded admission, rather than copy weighting alone, drives this improvement. P models less copy than the weighted baseline (W) and the owner baseline (O) on both opportunity workloads, and is materially faster than W and O on Cholesky and than O on LU; its 1.020\(\times\) edge over W on LU is competitive, below the 1.03\(\times\) materiality threshold. On CG, W changes the placement yet leaves modeled copy at 42.95 GB and runs 1.841\(\times\) slower than P, showing that unbounded copy weighting fails to capture the same locality.

\emph{Timing controls.} The three unchanged workloads bound the timing noise present in the effective comparisons. FDTD, CG, and MiniWeather produce identical H, P, and no-prefix (N) placement digests and modeled-copy totals, so their P/H ratios measure scheduler-path and system variation alone, and FDTD and MiniWeather are identical across all six variants. FDTD's P/H ratios span 0.801--1.157\(\times\) under identical decisions, which quantifies the timing noise of its short benchmark interval.

\emph{Gate removal.} Removing the prefix gate (N) exposes both sides of the safety--opportunity trade-off. On Cholesky, N is 1.161\(\times\) faster than P across all six pairs and lowers modeled copy to 5.90 GB. On LU, P is instead 1.210\(\times\) faster than N across all six pairs, and N accumulates 732.3 ms of modeled-prefix damage while lowering modeled copy to 3.88 GB. The same admission rule rejects an aggressive placement on Cholesky while preventing a harmful one on LU, illustrating the intended condition-dependent behavior.

\section{Related Work}\label{sec:related-work}

\textbf{HEFT and bounded lookahead.} HEFT places each task on the processor of minimum earliest finish time, accounting for the communication that delays its inputs~\citep{topcuoglu2002heft}, and later work enriches this decision with information about the future graph. PEFT adds an optimistic future-cost table~\citep{arabnejad2014peft}, lookahead HEFT evaluates a placement's effect on child tasks~\citep{bittencourt2010lookahead}, and interference-aware scheduling jointly models computation- and I/O-induced flows when constraining placement~\citep{bleuse2018anticipation}. These methods improve placement by predicting or searching the unscheduled graph. Unlike these approaches, prefix-safe placement refines the current HEFT decision rather than predicting future task behavior. It admits a lower-copy candidate device only when the resulting placement remains within a bounded tolerance of the modeled completion of already committed work.

\textbf{Task runtimes and locality-aware scheduling.} Task-based runtimes couple placement with data movement, and a broad line of work makes that coupling locality-aware. StarPU's data-aware scheduler combines software-managed shared memory with transfer-aware placement~\citep{augonnet2010dataaware,augonnet2011starpu}, while PaRSEC~\citep{bosilca2013parsec}, Legion~\citep{bauer2012legion}, and CUDASTF~\citep{augonnet2024cudastf} expose abstractions for dependencies, placement, and data management. On top of such runtimes, XKaapi's DADA schedules by data affinity~\citep{bleuse2014dada}, LAHeteroprio quantifies locality's effect on task-based performance~\citep{bramas2019lahp}, MCL selects resources by locality~\citep{kamatar2020locality}, and comparative studies place HEFT beside data- and locality-aware multi-CPU/multi-GPU schedulers~\citep{lima2015strategies}. More recent systems optimize locality under GPU-memory limits~\citep{gonthier2023taming}, combine ready-task locality with eviction for GPU and out-of-core linear algebra~\citep{gonthier2025darts}, and rebalance already-mapped work across GPUs~\citep{john2024multigpu}. These schedulers pursue locality through affinity heuristics, resource selection, or work migration. Unlike these schedulers, our work does not replace the underlying placement policy. Instead, it refines the HEFT decision by admitting lower-copy alternatives only when they preserve the modeled completion of already committed work.

\textbf{Transfer attribution.} Attributing GPU data movement to program context is usually a profiling task; HPCToolkit, for instance, attributes kernels and copies to heterogeneous calling contexts~\citep{zhou2021hpctoolkit}. Our runtime-issued transfer stream adds a complementary, decision-level view: it links each requested peer transfer to the scheduler decision and logical-data identity that produced it, connecting a placement choice directly to the data movement it induces.

\section{Limitations}\label{sec:limitations}

\textbf{Guarantee.} Prefix-safe placement constrains only the modeled completion of work that has already been committed. It does not predict future task placements, identify critical paths, or guarantee globally optimal makespan or communication. The scheduler therefore provides a bounded refinement of the current HEFT decision rather than a global optimization procedure.

\textbf{Evaluation scope.} The admission rule is defined independently of workload identity, but the present evaluation is limited to the workloads, platform, and runtime studied here. The five-workload matrix demonstrates the scheduler's behavior in both placement--residency opportunity cases and identical-placement controls, but it does not estimate the prevalence of such opportunities in broader applications. The experiments further assume deferred task submission on a homogeneous eight-GPU node and do not evaluate dynamic task arrival, heterogeneous devices, or comparisons with external runtime schedulers.

\textbf{Implementation and measurement.} The prototype uses a simplified transfer-cost model: a single effective bandwidth and a shared GPU-0 task-cost calibration, omitting pairwise NODE/SYS bandwidth differences, contention, copy--compute overlap, and memory-capacity effects such as eviction. The reported copy quantities are therefore scheduler-model outputs rather than profiler-measured device traffic, and the controlled-trace attribution rests on the task/data/fixed-successor join rather than an independent runtime version epoch; confirming physical PCIe/NVLink traffic would require CUPTI activity tracing or an equivalent profiler~\citep{nvidia2026cupti}. The implementation also prioritizes clarity over scheduler efficiency, leaving lower scheduler overhead and validation on larger systems and broader workloads as future work.

\section{Conclusion}\label{sec:conclusion}

Task placement in multi-GPU task-graph runtimes influences both execution time and future data residency. This paper identifies the resulting placement--residency conflict: although HEFT-style scheduling minimizes the modeled finish time of the current task, the selected placement may increase the communication required by later dependent tasks. To address this problem, we present prefix-safe copy-aware placement, a lightweight refinement of HEFT that evaluates lower-copy alternatives relative to the HEFT baseline and admits them only when they remain within a bounded tolerance of the modeled completion of already committed work. A prototype implementation in CUDASTF demonstrates that this bounded refinement consistently exploits placement--residency opportunities without changing HEFT behavior when no admissible opportunity exists. Across the qualified counterbalanced evaluation, the proposed scheduler reduces scheduler-modeled copy and improves performance on the workloads where placement changes occur, while the gate-removal ablation shows that copy reduction alone is insufficient and that the benefit of deviating from the HEFT baseline depends on the observed runtime state. The current implementation is limited to the workloads and platform evaluated in this study, and extending the approach to broader runtime environments, independently selected workloads, dynamic task arrival, external runtime baselines, and lower scheduler overhead remains future work. Nevertheless, the results show that bounded HEFT-relative locality refinement provides a practical mechanism for improving data locality while preserving the performance robustness of HEFT.
